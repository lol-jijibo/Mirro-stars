🪞 Mirro — AI 智能问题解决引擎
项目简介
Mirro 是一个面向年轻人（18-35岁）的 AI 驱动问题解决平台。用户输入自己的困惑或问题后，系统会自动完成：AI 分类 → 全网搜索真实案例 → LLM 生成结构化解答 → Mermaid 方案流程图 → 分步执行计划，全流程通过 SSE 流式推送实时呈现，给用户带来"AI 正在为你思考"的沉浸式体验。

与传统的 AI 问答不同，Mirro 不仅仅是"一问一答"，而是将每个问题当作一个需要被解决的项目来对待 —— 它会判断问题类型（行动型 vs 认知型），行动型问题会输出完整的流程图和执行步骤，认知型问题则提供多角度的深度分析。同时，每个答案都附有来自互联网的真实参考案例来源，增强答案的可信度。

技术栈： Vue 3 + TypeScript + Tailwind CSS v4 + Vite（前端） · Python FastAPI + SSE（后端） · OpenAI/Claude/DeepSeek 多模型兼容 · Tavily 全网搜索 · MySQL · ECharts + Mermaid.js 可视化

代码规模： ~2800 行（后端 928 行 Python + 前端 1828 行 Vue/TS），结构清晰，注释完备。

✨ 核心亮点
1. 智能双类型答案策略（action vs insight）
这是 Mirro 最核心的设计决策。AI 在生成答案之前，先判断用户问题的本质类型：

行动型（action）： 转行规划、学习路线、健身计划等 → 输出正文 + 可执行步骤 + 流程图
认知型（insight）： 情感困惑、职业迷茫、人生选择等 → 只输出深度分析正文，不画蛇添足
这避免了"用户问情感问题结果收到一张流程图"的尴尬，让 AI 的回答方式适配问题本质。

2. SSE 多阶段流式推送
答案不是一次性返回的，而是通过 8 个阶段逐步推送：


category → searching → sources → type → content（逐段） → flowchart → steps → done
前端根据事件类型分区域渲染 —— 分类标签先出来，然后搜索来源出现，接着正文像打字机一样逐段显示，最后流程图和步骤卡片依次就位。每个阶段独立渲染，互不阻塞，用户体验流畅自然。

3. 全网案例搜索 + AI 引用
提交问题后，系统自动通过 Tavily Search API 在全网搜索相关经验分享和解决方案，这些真实案例既作为 LLM 回答的参考素材，也作为答案的引用来源展示给用户。即使搜索服务宕机，核心的 AI 答案生成也不受影响 —— 搜索是加分项，不是生命线。

4. 方案流程图可视化（Mermaid.js）
当 AI 判断问题是行动型且有明确的决策路径时，会生成 Mermaid 流程图语法，前端通过 Mermaid.js 实时渲染为可视化图形。流程图节点紧扣用户的具体问题，从"当前处境"出发，经过 3-6 个关键决策点，到达"预期成果"。

5. 时间轴式分步执行计划
每个行动型答案都附带 3-6 步的可执行计划，以时间轴卡片形式展示。每步包含：

阶段名称（如"准备期"、"行动期"）
具体要做什么 + 为什么要这么做
预计耗时（如"1-2 周"、"3 个月"）
用户可以按顺序执行，像跟随一个私人教练完成目标。

6. 统计看板（ECharts 可视化）
内置数据分析看板，用饼图展示问题分类分布，用折线图展示每日提问趋势，顶部还有总提问数/已解答/解决率的概览卡片。所有图表按需加载 ECharts 组件，最小化打包体积。

7. 精致的 UI 设计细节
6 色渐变卡片轮换系统： 问题列表和历史记录使用 6 套不同配色的渐变卡片（靛蓝/天蓝/翠绿/琥珀/玫红/紫色），每张卡片有独特的 shadow、hover 光晕和动画效果
骨架屏加载： 每种页面状态都有对应的 Skeleton Loader，消除白屏等待的焦虑
fadeInUp 交错动画： 列表项按序渐入，视觉节奏感强
流式打字机效果： Markdown 段落逐段追加渲染，配合闪烁光标
回车快捷提交 + 实时字数统计： 输入体验打磨到位
8. 工程化沉淀
LLM 调用自带指数退避重试： 3 次重试、退避倍率 1.5×（1s → 1.5s → 2.25s），智能区分可重试错误（5xx/超时/连接）和不可重试错误（4xx）
多模型提供商兼容： 通过 LLM_BASE_URL 配置兼容 OpenAI / Claude / DeepSeek / 通义千问等任何兼容 OpenAI 接口的服务
MySQL 兼容适配层： 内部自动将 SQLite 的 ? 占位符转换为 MySQL 的 %s，业务代码零改动
自动建表 + 兼容旧表： 首次启动自动创建表结构，检测缺失列时自动 ALTER TABLE 补齐
按需加载路由 + ECharts 组件： 减小首屏打包体积
9. 前后端分离的清晰架构

后端（Python FastAPI）
 ├── api/          # 路由层：SSE流式端点、CRUD接口
 ├── services/     # 服务层：AI生成、搜索聚合、流程图验证
 ├── models/       # 数据层：Pydantic校验 + MySQL封装
 └── core/         # 配置层：环境变量统一管理

前端（Vue 3 + TypeScript）
 ├── views/        # 4个页面：首页提问、答案详情、历史列表、统计看板
 ├── components/   # 7个组件：输入框、流式渲染、流程图、步骤卡片、来源列表等
 ├── stores/       # Pinia状态：SSE流式数据缓存
 ├── api/          # SSE客户端 + HTTP请求封装
 └── router/       # SPA路由 + 动态标题守卫
📊 核心数据流

用户输入问题
    ↓
POST /api/questions (SSE)
    ↓
[阶段1] AI 分类 → 前端展示分类标签
[阶段2] Tavily 全网搜索 → 前端展示参考来源
[阶段3] LLM 生成答案 → 前端打字机渲染 Markdown
[阶段4] 流程图生成 → 前端 Mermaid 渲染
[阶段5] 步骤计划 → 前端时间轴卡片
[阶段6] 持久化到 MySQL → 完成，跳转答案页
    ↓
用户可随时回看历史、查看统计看板
